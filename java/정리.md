## 동일성과 동등성
동일성은 '두 인스턴스가 같은 존재인지 판단'하는 것  
자바에서는 `==` 연산자로 두 인스턴스의 레퍼런스 변수를 비교함  
비교의 결과가 true라면 동일한 인스턴스  
동등성은 '두 인스턴스가 같은 값으로 평가되는지 확인'하는 것
String 클래스는 자바의 `equals()` 메서드를 통해 확인하고, 직접 선언한 클래스는 equals() 메서드를 오버라이딩해 어떤 조건에서 동등하다고 판단할지 정해줘야 함

## 객체지향
객체지향의 핵심은 `추상화`와 `다형성`  
추상화와 다형성은 인터페이스로 연결되는데  
인터페이스 없이 개발을 하면 클라이언트 코드가 구체적인 클래스에 의존하게 되고, 하나의 클래스가 변경되면 다른 클래스도 변경해줘야 하는 문제가 생긴다.  
이런 문제를 해결하려면 클라이언트 코드가 구체적인 클래스가 아니라 인터페이스에 의존해야한다.  
결국 클라이언트 코드와 구체적인 클래스 모두 인터페이스에 의존하게 된다.  

몇 가지 이점으로 첫 번째. `클라이언트 코드는 해당 인터페이스를 구현하고 있는 구체적인 클래스의 세부 구현을 몰라도 된다.`  
인터페이스에 의존하고 있기 때문에 클래스가 바뀌더라도 인터페이스가 바뀌지 않는 한 클라이언트 코드로 변경 내용이 전파되지 않는다.  
두 번째. `클라이언트 코드는 해당 인터페이스를 구현하고 있는 구체적인 클래스가 구현되지 않았더라도 자신의 코드를 개발할 수 있다.`  
테스트 코드를 만들어 해당 인터페이스가 해야 하는 역할에 대해 정의해 주기만 하면 됨.  

다형성에 대해서는 인터페이스의 레퍼런스 변수는 해당 인터페이스를 구현하는 여러 클래스들의 인스턴스를 담을 수 있다. 레퍼런스 변수 안에 어떤 인스턴스가 들어있는지 신경쓰지 않고 메서드를 호출하면 안에 들어 있는 인스턴스 종류에 따라 완전히 다른 동작이 실행될 수 있다.
만약 인터페이스가 필수로 구현해야할 메서드의 집합 역할만 했다면, 인터페이스를 선언해도 추상화했을 때의 이점을 누릴 수 없게 될 것.

## 프로토콜
통신 규약으로 일종의 약속  
사람도 같은 언어를 사용해야 의사소통이 가능한 것처러머, 네트워크 역시 보내는 쪽과 받는 쪽이 서로 동일한 형식과 규칙으로 데이터를 주고 받아야 통신이 가능

## 웹 브라우저에서 www.google.com에 접속하면 어떤 일이 생길까
사용자의 웹 브라우저에서 www.google.com을 입력하면 해당 요청은 DNS에 www.google.com이라는 도메인이 어떤 IP주소를 가지고 있는지 묻는다.  
DNS에서 IP 주소를 알려주면 트래픽이 구글 웹 서버를 향해 출발한다.  
그럼 네트웨크 상의 라우터, 스위치 등 여러 네트워크 장비를 거쳐 구글의 웹 서버에 도달한다.  
구글 웹 서버에서 해당 요청에 맞는 HTML 파일을 응답으로 돌려주면, 그 HTML 파일은 왔던 길을 돌아가서 클라이언트에게 도착해 사용자의 웹 브라우저에 렌더링된다.

### 웹 브라우저에서 URL을 처리하는 과정을 설명, URL이 www.google.com/search?q=url과 같이 되어 있다면?
먼저 브라우저가 URL을 파싱해서 도메인인 www.google.com, 경로인 /search, 쿼리 파라미터인 q=url을 해석합니다.  
그다음 DNS 조회를 통해 도메인을 IP 주소로 변환하고, 서버와 TCP 연결을 맺습니다.  
HTTPS라면 TLS 핸드셰이크로 암호화 통신을 준비한 뒤 GET /search?q=url 형태의 HTTP 요청을 보냅니다.  
서버는 검색 결과 HTML을 응답하고, 브라우저는 HTML, CSS, JavaScript를 파싱하고 렌더링해서 최종 화면을 사용자에게 보여줍니다.

### 만약 DNS가 www.google.com 이라는 도메인에 대한 정보를 가지고 있지 않다면?
만약 DNS가 www.google.com에 대한 정보를 바로 가지고 있지 않다면,
DNS 서버는 직접 최종 답을 아는 서버를 다시 찾아가는 과정을 거치게 됩니다.  
보통 클라이언트는 먼저 로컬 DNS 서버에 질의하고, 로컬 DNS 서버에 캐시가 없으면 루트 DNS 서버에 물어봅니다.  
루트 DNS 서버는 google.com을 관리하는 곳의 정보를 알려주고, 그다음 .com TLD DNS 서버에 질의합니다.  
.com DNS 서버는 다시 google.com의 권한 있는 DNS 서버(Authoritative DNS Server) 정보를 알려줍니다.  
마지막으로 권한 DNS 서버에 질의해서 www.google.com의 실제 IP 주소를 받아옵니다.  
즉, DNS가 처음부터 답을 모르더라도 단계적으로 상위 DNS 서버들을 따라가면서 최종 IP 주소를 찾아오는 구조입니다.

### 트래픽은 사용자의 PC에서 구체적으로 어떤 과정을 거쳐 빠져나가는지 OSI 7계층이나 TCP/IP 4계층을 기준으로 설명
사용자 PC에서 트래픽이 나갈 때는, 애플리케이션에서 만든 데이터가 각 계층을 지나면서 캡슐화됩니다.  
예를 들어 브라우저가 HTTP 요청을 만들면, 전송 계층에서 TCP 헤더가 붙어 세그먼트가 되고, 인터넷 계층에서 IP 헤더가 붙어 패킷이 됩니다.  
이후 네트워크 인터페이스 계층에서 MAC 주소가 포함된 프레임으로 만들어지고, 마지막으로 NIC가 이를 전기 신호나 무선 신호로 변환해서 네트워크로 내보냅니다.  
반대로 수신 측에서는 하위 계층부터 헤더를 제거하면서 원래 데이터로 복원합니다.

### 라우터는 어떻게 구글 웹 서버를 찾아가나요?
라우터는 google.com이라는 도메인 이름을 보고 찾는 것이 아니라, 이미 DNS 조회를 통해 알아낸 목적지 IP 주소를 기준으로 패킷을 전달합니다.  
즉 사용자의 PC가 먼저 DNS를 통해 www.google.com의 IP 주소를 알아내고, 그다음 패킷 안에 출발지 IP와 목적지 IP를 넣어 전송합니다.  
라우터는 이 패킷을 받으면 목적지 IP 주소를 확인하고, 자신이 가지고 있는 라우팅 테이블을 조회해서 그 목적지 대역으로 가기 위한 다음 홉(next hop) 또는 출력 인터페이스를 결정합니다.  
그리고 그다음 라우터로 패킷을 넘기고, 이 과정이 인터넷 구간에서 반복되면서 최종적으로 구글 웹 서버가 속한 네트워크까지 도달하게 됩니다.  
즉, DNS는 목적지 IP를 알아내는 역할, 라우터는 그 IP를 보고 어디로 보낼지 결정하는 역할입니다.


### 구글 웹 서버 장비에서 어떻게 트래픽이 웹 서비스로 접근하나요?
구글 웹 서버 장비에 트래픽이 도착하면, 먼저 패킷은 서버의 NIC(Network Interface Card) 로 들어옵니다.  
NIC는 전기 신호나 광 신호를 디지털 데이터로 바꾸고, 이를 운영체제의 네트워크 스택에 전달합니다.  

그다음 운영체제는 이더넷 프레임의 MAC 헤더를 확인하고, 자기 장비로 온 프레임이면 2계층 헤더를 제거합니다.  
이후 IP 헤더를 확인해서 목적지 IP가 서버 자신인지 검사하고, 맞다면 상위 프로토콜이 TCP인지 UDP인지 확인해서 전송 계층으로 넘깁니다.  

TCP라면 운영체제는 목적지 포트 번호를 보고 어떤 프로세스가 해당 포트를 열어두고 있는지 확인합니다.  
예를 들어 웹 서비스가 80번 포트나 443번 포트에서 listen 중이라면,
그 소켓으로 데이터를 전달합니다.  

그러면 웹 서버 프로세스나 애플리케이션 서버가 HTTP 요청을 읽고, 필요한 비즈니스 로직을 수행한 뒤 응답을 생성합니다.
생성된 응답은 다시 반대로 TCP/IP 스택을 타고 내려가서 패킷으로 만들어져 네트워크로 전송됩니다.  

즉, NIC로 들어온 패킷이 OS 네트워크 스택에서 역캡슐화되고, 포트 번호를 기준으로 웹 서비스 프로세스에 전달되어 처리된다고 정리할 수 있습니다.

### 사용자의 웹 브라우저에 HTML이 렌더링되는 것 외에 다른 과정이 또 있을까요?
브라우저는 서버로부터 HTML을 받은 뒤 단순히 화면에 그리는 것뿐 아니라, 그 안에 포함된 CSS, JavaScript, 이미지, 폰트 같은 추가 리소스도 다시 요청합니다.  

또한 HTML을 파싱해 DOM을 만들고, CSS를 파싱해 CSSOM을 만든 뒤, 이를 결합해서 렌더 트리를 만들고 레이아웃과 페인팅 과정을 수행합니다. 이후 JavaScript가 실행되면서 DOM이 변경될 수 있고, 추가적인 비동기 API 호출(fetch, AJAX) 이 발생할 수도 있습니다.  

즉, 웹 브라우저는 단순히 HTML을 표시하는 것이 아니라 추가 리소스 로딩, 스크립트 실행, 이벤트 처리, 네트워크 재요청, 캐시 활용, 쿠키/세션 처리까지 포함한 여러 과정을 수행합니다.

## TCP와 UDP의 차이
TCP와 UDP는 둘 다 전송 계층에 속하는 프로토콜입니다.
그러나 트래픽을 전송하는 방식에 있어 두 프로토콜은 큰 차이를 보임.

TCP의 경우 트래픽을 보낼 때 패킷이라는 단위를 사용.  
이 패킷의 전송 과정 전후로 연결을 맺고 끊는 과정이 포함되고, 전송 도중에도 올바르게, 순서대로 도달했는지를 확인하는 과정이 있다.
그래서 패킷 전송시 신뢰성이 더 높다.  

반면 UDP의 경우 트래픽을 보낼 때 데이터그램이라는 단위를 사용하며, 전송 도중에 TCP처럼 확인하는 과정이 없다.  
따라서 상대적으로 TCP보다 신뢰성은 낮지만 속도는 빠르다.

## 공인 IP와 사설 IP의 차이
공인 IP는 전 세계 인터넷 어디에서도 접근 가능한 유일한 주소.
IPv4를 기준으로 사용할 수 있는 주소는 대략 43억 개 정도인데, 인터넷이 연결된 모든 기기에 IP 주소를 부여하기엔 수가 너무 부족합니다.  
그래서 IPv6처럼 더 많은 주소를 할당할 수 있는 방식이 등장했지만 아직 널리 사용되고 있지는 않고, 사설 IP가 IP 부족 문제를 부분적으로나마 해결해주고 있습니다.  
사설 IP는 어떤 네트워크 망 내부에서만 사용되는 IP 주소로, 보통 192.168로 시작하며 우리가 가장 흔히 볼 수 있는 IP이기도 함.  
사설 IP를 사용하는 이유는 공인 IP 할당에 많은 비용이 들어간다는 이유도 있지만, 네트워크 망을 외부와 분리하여 해킹 등의 위험으로부터 상대적으로 안전하게 인프라를 구성하려는 목적도 있습니다.


## 로드 밸런싱
하나의 서버가 받아야 할 트래픽을 여러 서버가 나눠 받도록 부하를 분산하는 것.  
부하 분산을 위해서 L4, L7 스위치 같은 네트워크 장비나 엔진엑스 같은 소프트웨어를 사용하여 서버를 구성합니다.  
이렇게 되면 부하가 분산되는 효과 외에도 스케일 아웃 하기 좋은 환경이 됩니다.
(스케일 아웃: 서버 한 대의 성능을 키우는 게 아니라, 서버 대수를 늘려서 처리량을 늘리는 것)
또한 수평적으로 확장이 가능하기 때문에 들어오는 트래픽이 늘어나는 상황에 대응하기 좋다는 장점이 있습니다.

## 스프링 프레임워크와 스프링 부트의 차이
스프링 부트는 스프링 프레임워크를 사용하기 쉽도록 만들어 줍니다.  
스프링 프레임워크는 객체지향적인 개발을 위해 필요한 의존성 주입이나 제어의 역전을 손쉽게 활용할 수 있어 애플리케이션에서 핵심적인 로직에 집중하도록 도와준다.  
스프링 프레임워크는 XML과 같은 여러 가지 초기 설정을 해줘야 하는데, 스프링 부트를 쓰면 초기 설정을 간단하게 할 수 있고 내장 Tomcat을 사용하면 애플리케이션을 손쉽게 빌드하고 배포할 수 있다.

### 객체지향적인 개발이 어떤 것인지 설명
객체지향적인 개발이란 각 객체가 자신의 역할과 책임을 가지도록 설계하는 개발 방식.  
단순히 클래스를 나누는 것이 아니라, 데이터와 기능을 따로 두지 않고 객체가 자신의 상태와 동작을 함께 관리하도록 만드는 것이 중요하다고 생각.  

그래서 한 클래스에 로직이 몰리지 않게 하고, 필요한 책임을 적절히 분리해서 객체들이 서로 협력하도록 구성하는 것이 객체지향적인 개발이라고 생각.  
이렇게 설계하면 요구사항이 바뀌었을 때도 영향을 덜 받게 되고, 유지보수나 확장도 더 쉬워진다고 생각.  
예를 들어 주문 시스템이라면 주문, 할인, 결제 같은 역할을 각각 나눠서, 각 객체가 자기 역할에 맞는 기능을 담당하도록 만드는 방식이 객체지향적인 개발이라고 생각.

### 의존성 주입
객체가 필요한 다른 객체를 직접 생성해서 사용하는 것이 아니라, 외부에서 주입받도록 해서 구체 클래스가 아니라 인터페이스에 의존하게 만드는 방식이라고 생각.  
객체지향에서는 클라이언트 코드가 구현체에 직접 의존하면 한 클래스가 바뀔 때 다른 클래스까지 같이 수정해야 하는 문제가 생길 수 있는데, 의존성 주입을 사용하면 이런 결합을 줄일 수 있습니다.  
예를 들어 클라이언트는 인터페이스만 알고 있고, 실제 어떤 구현체를 사용할지는 외부에서 넣어주기 때문에 구현이 바뀌어도 클라이언트 코드는 크게 수정하지 않아도 됩니다.  
결국 의존성 주입은 추상화와 다형성을 더 잘 활용해서 변경에 유연하고 테스트하기 쉬운 구조를 만드는 방법이라고 생각합니다.

### 제어의 역전
보통 객체가 직접 필요한 객체를 생성하고 관리할 수도 있는데, IoC를 적용하면 이런 제어를 스프링 컨테이너 같은 외부가 대신 맡게 됨.  
그래서 개발자는 객체 생성이나 연결보다는 핵심 로직에 더 집중할 수 있고, 의존성 주입을 통해 객체 간 결합도도 낮출 수 있다고 생각합니다.

### 라이브러리와 프레임워크 차이
라이브러리는 내가 필요할 때 가져다 쓰는 도구이고, 내 코드가 실행 흐름을 제어.  
프레임워크는 애플리케이션의 전체 실행 흐름을 관리하고, 프레임워크가 내 코드를 호출한다.
특정 프레임워크로 개발된 애플리케이션을 다른 프레임워크를 바꾸는 것은 어렵다 왜냐면 실행의 흐름이 해당 프레임워크에 맞춰 개발되어 있기 때문.  
라이브러리는 상대적으로 다른 라이브러리로 바꾸는 것이 간단. 왜냐면 작성한 코드에서만 부분적으로 사용되고 있기 때문

### maven과 gradle 차이
Maven은 pom.xml과 lifecycle 중심이라 구조가 정형화되어 있고 협업 시 이해하기 쉬운 장점이 있고,  
Gradle은 task 기반에 증분 빌드와 캐시 같은 최적화 기능이 있어서 대형 프로젝트나 멀티모듈 환경에서 더 유연하고 성능상 유리할 수 있다고 생각합니다.  
그래서 단순하고 표준적인 프로젝트면 Maven, 커스터마이징이나 빌드 최적화가 중요하면 Gradle을 선택할 수 있다고 봅니다.  
정리하면 Maven은 표준화와 안정성, Gradle은 유연성과 빌드 최적화 쪽이 더 강합니다.

### URL과 URI의 차이
URI는 URL보다 상위 개념으로, 리소스를 식별하는 통합 개념입니다.  
URI에는 URL과 URN이 포함됩니다.
URL(Uniform Resource Locator)은 리소스의 위치를 나타내며, scheme, host, port, path 등을 통해 자원에 어떻게 접근할지를 표현합니다.  
URN(Uniform Resource Name)은 리소스의 고유한 이름을 나타내고 책 번호로 사용되는 ISBN이 있습니다.  
즉, URI는 리소스를 식별하는 개념이고, URL은 그중 위치를 이용해 식별하는 방식이며, URN은 이름을 이용해 식별하는 방식입니다.


### 웹 브라우저가 웹 서버와 상호작용 할 수 있는 방법에는 어떤 것들이 있나
URL을 통해 접속하는 방법과 form 태그를 사용하여 상호작용하는 방법이 있습니다. 두 방법은 모두 사용자의 웹 페이지를 새로고침을 해야한다는 한계가 존재합니다.  
그래서 AJAX로 요청하는 방법이 있습니다. AJAX로 요청하면 웹 페이지를 새로고침 하지 않고도 서버와 상호작용 할 수 있습니다.
앞서 이야기한 것들은 클라이언트 쪽에서 서버로 요청을 해야만 데이터를 보낼 수 있었고, 서버에서 클라이언트로 보내는 방법이 필요했고, 그 대안으로 웹 소켓을 통한 연결 정보를 주고 받는 것이 있습니다.
